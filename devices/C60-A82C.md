# Device C60-A82C

## Overview

- Detected via `btproxy list`.
- Device Name (`0x2A00`): `C60-A82C`
- Generic Access service is present.
- Heart Rate service is present, so the device likely exposes heart-rate data via notifications.
- A vendor-specific service `0000ff00-0000-1000-8000-00805f9b34fb` is present.
- A custom service `00010203-0405-0607-0809-0a0b0c0d1912` is present.

## Readable Characteristics

### `0x2A00` Device Name

- UUID: `00002a00-0000-1000-8000-00805f9b34fb`
- Properties: `READ NOTIFY`
- Raw value: `43 36 30 2D 41 38 32 43 00 00 00 00 00 00 00 00 00 00`
- ASCII decoding: `C60-A82C`
- Interpretation: standard BLE Device Name. This appears to be the human-readable identifier broadcast by the device.
- Notify behavior observed via `btproxy n`: subscription attempt fails immediately with `ERROR: Failed to enable notify`. So the characteristic is exposed as `NOTIFY` by the discovery listing, but the server/device combination does not currently allow enabling notifications on it.

### `0x2A01` Appearance

- UUID: `00002a01-0000-1000-8000-00805f9b34fb`
- Properties: `READ`
- Raw value: `00 00`
- Little-endian uint16: `0`
- Interpretation: standard BLE Appearance. Value `0` usually means `Unknown Appearance`, so it does not identify a specific device class.

### `0x2A04` Peripheral Preferred Connection Parameters

- UUID: `00002a04-0000-1000-8000-00805f9b34fb`
- Properties: `READ`
- Raw value interpreted as uint16 little-endian: `[30, 50, 20, 600]`
- Interpretation:
    - Minimum connection interval: `30` units = `37.5 ms`
    - Maximum connection interval: `50` units = `62.5 ms`
    - Slave latency: `20`
    - Supervision timeout: `600` units = `6000 ms`
- This is a standard BLE parameter block and appears internally consistent.

### Custom readable characteristic

- Service: `00010203-0405-0607-0809-0a0b0c0d1912`
- UUID: `00010203-0405-0607-0809-0a0b0c0d2b12`
- Properties: `READ WRITE_NO_RESPONSE`
- Raw value: `00`

- Interpretation: currently not enough information. The value may be a boolean flag, a status byte, or an uninitialized placeholder.

## Notify Exploration Summary

- `0x2A00`: declared as `NOTIFY`, but notification enable failed immediately.
- `0xFF01`: subscription works, but no spontaneous payloads were observed during a short idle capture window. It does emit replies when commands are written to `0xFF02`.
- `0x2A37`: subscription works, but no heart-rate packets were observed during a short idle capture window.
- Practical conclusion: the two useful notify channels are `0xFF01` and `0x2A37`, but both appear event-driven rather than chatty at idle.

## Proprietary Notification Protocol

### Transport

- Command write characteristic: `0000ff02-0000-1000-8000-00805f9b34fb`
- Reply / ACK notify characteristic: `0000ff01-0000-1000-8000-00805f9b34fb`

### Checksum

- The APK computes the last byte of each frame with this rule:
    - sum all preceding bytes modulo 256
    - checksum = `((sum * 0x56) + 0x5A) & 0xFF`

### Message Frames

#### `setMessageType(appType)`

- Purpose: declares the source app icon/type that the bracelet should show for the incoming notification.
- Frame format:
    - `0A 02 00 00 <appType> <checksum>`

#### `setMessage2(kind, text)`

- Purpose: sends a text payload.
- `kind = 0x01`: title / sender
- `kind = 0x02`: body / content
- Frame format:
    - byte `0`: `0x0A`
    - bytes `1-2`: payload length in little-endian, where payload length is `1 + len(textBytes)`
    - byte `3`: `kind`
    - bytes `4..`: UTF-8 bytes of the text
    - last byte: checksum
- The app splits long frames into 20-byte chunks before writing them.
- The app truncates text before framing:
    - title: about `0x20` bytes max
    - body: about `0x80` bytes max

#### `END_MESSAGE`

- Final marker frame after title and body:
    - `0A 01 00 03 0E`

### ACK Sequence On `FF01`

- The app waits for a notify on `FF01` after each stage.
- Every ACK starts with byte `0x8A`.
- The fourth byte identifies the completed stage:
    - `0x00`: ACK for `setMessageType`
    - `0x01`: ACK for title
    - `0x02`: ACK for body
    - `0x03`: ACK for `END_MESSAGE`

### Full Delivery Flow

1. Write `setMessageType(appType)` to `FF02`
2. Wait for `FF01` ACK with stage `0x00`
3. Write title frame `setMessage2(1, title)` to `FF02`
4. Wait for `FF01` ACK with stage `0x01`
5. Write body frame `setMessage2(2, text)` to `FF02`
6. Wait for `FF01` ACK with stage `0x02`
7. Write `END_MESSAGE` to `FF02`
8. Wait for `FF01` ACK with stage `0x03`

## Live Validation With `btproxy`

- The protocol above was validated against the real bracelet, not only inferred from the APK.
- Device target used during testing: `A4:C1:38:9A:A8:2C`
- `FF01` listener was opened with `btproxy n a ff01 ...`
- The following write sequence succeeded end-to-end:

### 1. Set app type to WhatsApp

- Write to `FF02`:
    - `0A 02 00 00 07 BC`
- ACK received on `FF01`:
    - `8A 02 00 00 04 BA`

### 2. Send title `OpenAI`

- Write to `FF02`:
    - `0A 07 00 01 4F 70 65 6E 41 49 CE`
- ACK received on `FF01`:
    - `8A 02 00 01 04 10`

### 3. Send body `Test alert`

- Write to `FF02`:
    - `0A 0B 00 02 54 65 73 74 20 61 6C 65 72 74 A4`
- ACK received on `FF01`:
    - `8A 02 00 02 04 66`

### 4. Finish message

- Write to `FF02`:
    - `0A 01 00 03 0E`
- ACK received on `FF01`:
    - `8A 02 00 03 04 BC`

### Result

- This confirms that the bracelet accepts synthetic app notifications through the proprietary `FF02` / `FF01` protocol.
- The previous assumption that `FF02` was only a likely command channel is now obsolete: it is confirmed.
