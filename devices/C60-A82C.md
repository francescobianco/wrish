# Device C60-A82C

## Overview

- Detected via `btproxy list`.
- Device Name (`0x2A00`): `C60-A82C`
- MAC address used during testing: `A4:C1:38:9A:A8:2C`
- Firmware version string (from CMD_GET_DEVICE): `TB2C60T7`
- Generic Access service is present.
- Heart Rate service is present.
- A vendor-specific service `0000ff00-0000-1000-8000-00805f9b34fb` is present.
- A custom service `00010203-0405-0607-0809-0a0b0c0d1912` is present.
- Companion app package: `com.keephealthpro.android` (version 4.6.22)

## Authentication

No GATT authentication is required. The app follows this exact sequence on every connection:

1. Connect to device (BLE level)
2. Enable CCCD notifications on `FF01` (`char-write-req <cccd> 01 00`)
3. Send commands to `FF02`

The `SERVICE_PAIR` UUID (`0000ff04-0000-1000-8000-00805f9b34fb`) is used only as a BLE scan filter, not for GATT-level authentication. The `CMD_BIND_DEVICE` frame (`0c 04 00 00 05 06 07 c6`) is sent during initial pairing (once), not on every session.

## Startup Sequence (validated via BLE proxy, 2026-03-28)

The app sends this sequence immediately after connecting and enabling FF01 notifications:

```
43 00 00 dc   GET unknown (0x43)
01 00 00 b0   CMD_GET_DEVICE
02 00 00 06   CMD_GET_DEVICE_STATE
02 10 00 ...  CMD_SET_DEVICE_STATE  (write-back with 1 flag changed)
04 08 00 ...  CMD_SET_TIME          (sync clock)
03 07 00 ...  CMD_SET_USERINFO      (push user profile)
07 00 00 b4   CMD_GET_TARGE
27 00 00 74   CMD_GET_CURRENT_POWER (battery)
```

Then it fetches historical data for the past several days (activity, heart rate, blood pressure) and current health snapshot.

## Proprietary Command Protocol

### Transport

- Write characteristic: `0000ff02-0000-1000-8000-00805f9b34fb` (FF02)
- Reply / ACK notify characteristic: `0000ff01-0000-1000-8000-00805f9b34fb` (FF01)

### Frame Format

All commands follow this structure (confirmed from `CmdHelper.smali`):

```
[cmd] [len_lo] [len_hi] [payload...] [checksum]
```

- `cmd`: 1-byte command identifier
- `len_lo / len_hi`: little-endian uint16 length of the payload (bytes that follow, not counting checksum)
- `payload`: `len` bytes of command-specific data (may be empty)
- `checksum`: last byte, computed as:

```
sum = 0
for each byte in frame[0 .. -2]:      # all bytes except last
    sum = (sum + byte) & 0xFF          # truncate to byte at each step
checksum = ((sum * 0x56) + 0x5A) & 0xFF
```

Zero-payload commands are 4 bytes: `[cmd] 00 00 [checksum]`.

Response frames mirror the command byte with `0x80` OR-ed in (e.g. cmd `0x01` â†’ response `0x81`, cmd `0x27` â†’ response `0xA7`).

## Response Payloads (validated)

### CMD_GET_DEVICE (`0x01`) response

```
81 10 00  [firmware 8 bytes]  [hw]  [??]  [MAC 6 bytes]  [chk]
```

Example:
```
81 10 00  54 42 32 43 36 30 54 37  02  04  a4 c1 38 9a a8 2c  6e
          "T B 2 C 6 0 T 7"        hw       MAC address         chk
```

- Bytes [3..10]: firmware version string ASCII (`TB2C60T7`)
- Byte [11]: hardware version (`0x02`)
- Bytes [13..18]: MAC address in network byte order

### CMD_GET_CURRENT_POWER (`0x27`) response

```
a7 01 00  [battery%]  [chk]
```

- Byte [3]: battery percentage (0â€“100 decimal)

Example: `a7 01 00 43 4c` â†’ battery **67%**.

### CMD_GET_CURRENT_STEP (`0x20`) response

```
a0 0d 00  00  [steps uint32 LE]  [calories uint32 LE]  [distance uint32 LE]  [chk]
```

Example: `a0 0d 00 00 73 01 00 00 0c 00 00 00 03 01 00 00 d0`
- Steps: `73 01 00 00` = 371
- Calories: `0c 00 00 00` = 12 kcal

### CMD_GET_HART current snapshot (`21 01 00 00`) response

```
a1 05 00  00  [HR]  [diastolic]  [systolic]  [SpO2]  [chk]
```

Example: `a1 05 00 00 45 46 6c 62 04`

- Systolic = 0x6C = 108 mmHg

### Historical HR/BP/SpO2 (`21 05 00 01 ...`) response

Response header: `a1 [len_lo] [len_hi] 01 [year_lo year_hi mm dd] 0f [records...]`

Each record is **4 bytes**: `[HR] [diastolic] [systolic] [SpO2]`

One record per minute. Values of `0x00` indicate no measurement for that minute.

Example records from 2026-03-27:
```

5a 53 81 62  â†’ HR=90, BP=129/83, SpO2=98%
```

### CMD_GET_NOTICE / CMD_SET_NOTICE (`0x09`)

Query (zero payload): `09 00 00 60`
Response / write: `[89|09] 04 00 [byte0] [byte1] [byte2] [byte3] [chk]`

The 4-byte payload is a bitmask of enabled notification apps.
`ff ff ff ff` = all enabled.

Observed states (proxy log 2026-03-28):

| Event | Payload | Change |
|---|---|---|
| All enabled | `ff ff ff ff` | baseline |
| Twitter disabled | `fe ff 07 81` | byte[0] bit0 cleared |
| Twitter re-enabled | `ff ff 07 81` | byte[0] bit0 restored |

**Confirmed bit mapping (partial):**
- byte[0] bit 0 (`0x01`) = Twitter

The full bitâ†’app mapping requires one capture per app toggle. Bytes 2-3 changed to `07 81` independently of the Twitter toggle â€” other apps were already disabled in that session.

To enable all notifications (write to FF02): `09 04 00 ff ff ff ff 60`

## Notification Protocol

### Message Frames

#### `setMessageType(appType)`

- Purpose: declares the source app icon/type shown on the bracelet.
- Frame format: `0a 02 00 00 <appType> <checksum>`

#### `setMessage2(kind, text)`

- Purpose: sends a text payload.
- `kind = 0x01`: title / sender (max 32 UTF-8 bytes)
- `kind = 0x02`: body / content (max 128 UTF-8 bytes)
- Frame format:
    - byte `0`: `0x0A`
    - bytes `1-2`: `(len(textBytes) + 1)` as little-endian uint16
    - byte `3`: `kind`
    - bytes `4..`: UTF-8 text bytes
    - last byte: checksum
- Long frames are split into 20-byte chunks before writing.

#### `END_MESSAGE`

- Final marker frame: `0a 01 00 03 0e`

### ACK Sequence On `FF01`

Every ACK starts with byte `0x8A`. The fourth byte identifies the completed stage:

| Byte [3] | Stage |
|---|---|
| `0x00` | ACK for `setMessageType` |
| `0x01` | ACK for title |
| `0x02` | ACK for body |
| `0x03` | ACK for `END_MESSAGE` |

### Full Delivery Flow

1. Write `setMessageType(appType)` to FF02
2. Wait for FF01 ACK with stage `0x00`
3. Write title frame `setMessage2(1, title)` to FF02
4. Wait for FF01 ACK with stage `0x01`
5. Write body frame `setMessage2(2, body)` to FF02
6. Wait for FF01 ACK with stage `0x02`
7. Write `END_MESSAGE` to FF02
8. Wait for FF01 ACK with stage `0x03`

## Not Fully Explored Or Not Understood Yet

- `00010203-0405-0607-0809-0a0b0c0d2b12`: readable, but value `00` is too small to identify semantics.
- `0x2A37`: explored at idle; no payloads. Requires active heart-rate measurement.
- `0x2A05`: indication-only, no direct read attempted.
- `0x2A00`: readable and decodable, but notify enable fails despite characteristic advertising `NOTIFY`.
- `0x43`: seen in startup sequence but no response observed; purpose unknown.
- `0x25`: write command with 11-byte payload seen in proxy log; fields not decoded.
- `0x2E`: returns all `ff`; purpose unknown (possibly vibration/sleep schedule).
- `0x2C` raw ADC scale factor: the uint16 LE values are ADC counts; conversion to mmHg not determined.
- CMD_GET_DEVICE_STATE byte [14] flag: app writes it as `0x02` vs read value `0x00`; meaning unknown.
- CMD_SET_TIME trailing `00` byte: padding or sub-second field; not confirmed.
- CMD_SET_USERINFO gender byte: `0x00` observed; encoding (0=F/1=M or vice versa) not confirmed.
- Vibration configuration (`setVibrate`, header `0x70`): exact wire format not documented.
- ECG service (`0000ef00`), FFD service (`0000ffd0`), Active Upload service (`0000fc00`): present in UUIDs but not explored.
- Historical activity data format (`0x20` with date): minute-resolution step counts, exact byte layout not fully decoded.
